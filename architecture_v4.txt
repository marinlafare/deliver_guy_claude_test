1. YOUR BOSS'S QUESTION: DO WE NEED RL OR A HEAVIER ALGORITHM?
--------------------------------------------------------------------------------
  Short answer: NO. And here's the honest breakdown of why.

  1.1 REINFORCEMENT LEARNING — NOT YET, AND MAYBE NEVER

      RL needs:
        (a) A simulator OR a live environment where exploration
            (i.e., deliberately making BAD decisions to learn) is
            acceptable.
        (b) Hundreds of thousands of episodes to converge.
        (c) A reward signal that's attributable to individual actions.

      You have:
        (a) No simulator. Exploring in prod means real customers
            getting 60-min deliveries so the model can learn what
            not to do. Your NPS will crater.
        (b) 5,000 orders/day. Even if every order is an episode,
            convergence takes months — and the demand distribution
            shifts seasonally, so the model chases a moving target.
        (c) The reward for a single assignment is entangled with
            every OTHER assignment in the same tick. Rider A got
            order 1, which caused rider B to get order 2, which
            delayed rider C. Credit assignment is nightmarish.

      RL works beautifully for Uber/DoorDash at 10M+ orders/day
      with dedicated ML teams and simulation platforms. At 5k/day
      it's a science project that degrades your service while it
      learns.

      WHEN RL would make sense for Foody:
        - You've built a simulator from 6+ months of decision logs
          (we're collecting these — see v2 §9, v3 §10).
        - You've grown to 50k+ orders/day.
        - You have a dedicated data team.
      Until then: no.

  1.2 "HEAVIER" ALGORITHMS — HEAVIER ≠ BETTER

      Hungarian is O(n³) and gives the PROVABLY OPTIMAL assignment.
      You cannot beat it with a "heavier" algorithm — you can only
      match it at higher computational cost.

      What COULD improve results:
        - Better COST FUNCTION (better inputs to Hungarian), not a
          different solver.
        - Better BATCH FORMATION (feeding better candidate batches
          to the same solver).
        - Better PREDICTION (knowing when food will actually be
          ready, not just using the 15-min average).

      All three are input-quality problems, not algorithm problems.
      v4 focuses on these.

  1.3 WHAT WOULD ACTUALLY MOVE THE NEEDLE
      ┌──────────────────────────────────┬──────────────┬────────────┐
      │ Improvement                      │ Effort       │ Impact     │
      ├──────────────────────────────────┼──────────────┼────────────┤
      │ Better prep-time prediction      │ Low          │ HIGH       │
      │ Speculative pre-dispatch         │ Medium       │ HIGH       │
      │ Learned cost-function weights    │ Medium       │ MEDIUM     │
      │ Real-time route-aware ETAs       │ Medium       │ MEDIUM     │
      │ Full MILP solver                 │ High         │ LOW (~2%)  │
      │ Reinforcement learning           │ Very High    │ NEGATIVE   │
      └──────────────────────────────────┴──────────────┴────────────┘

      The top two items — prediction and pre-dispatch — are where
      this architecture invests.

  TO TELL YOUR BOSS: "Hungarian IS the optimal algorithm for this
  class of problem. The gains are in feeding it better data, not in
  replacing it. RL at our scale would make things worse, not better.
  Here's the math."

                    THE END-TO-END PIPELINE
              Optimized for total elapsed time (T_total)
================================================================================

  T_total = T_queue + T_prep + T_pickup_wait + T_travel_to_client

  Each segment has a different optimization lever. The pipeline is
  designed so that improvements to one segment don't regress another.

--------------------------------------------------------------------------------
3. STEP ① — ORDER PLACED
--------------------------------------------------------------------------------
  Trigger: client submits checkout.

  Actions:
    (a) Validate: seller is open, items available, client address
        within 2 km of seller. HARD FAIL if not.
    (b) Payment: authorize (not capture) the amount.
    (c) Create order record:
          state = PLACED
          placed_at = now
          seller_id, client_id, client_position, items
    (d) Estimate prep time: NOT a flat 15 min. Use a PER-SELLER
        prediction (see §4 below).
          food_ready_est = now + predicted_prep_time(seller, items)
    (e) Quote ETA to client:
          eta = food_ready_est + avg_pickup_wait + predicted_travel
        This is a PROMISE. The pipeline's job is to beat it.
    (f) Push order to seller app.
    (g) Emit event: OrderPlaced.

  State after: PLACED.
  T_queue optimization: ZERO. We push instantly to seller. No
  queueing, no batching of orders at this stage.

  Time budget consumed: 0 min. Clock starts.

--------------------------------------------------------------------------------
4. STEP ② — SELLER STARTS COOKING
--------------------------------------------------------------------------------
  Trigger: seller taps "Accept & Start" in seller app.

  Actions:
    (a) Record seller_ack_at. If no ack within 90s → alert ops,
        auto-call seller, offer client cancellation.
    (b) Update food_ready_est if seller provides one ("busy, 22 min
        today"). Seller's own estimate overrides the prediction.
    (c) Emit event: PreparationStarted.

  State after: PREPARING.

  ──────────────────────────────────────────────────────────────────
  PREP-TIME PREDICTION — THE SINGLE HIGHEST-IMPACT OPTIMIZATION
  ──────────────────────────────────────────────────────────────────
  Why: the 15-min average is a LIE. Some sellers are 8-min fast-food
  machines. Some are 25-min gourmet kitchens. And the SAME seller is
  faster at 11am than at 12:30 when they have 14 active orders.

  If you estimate prep as 15 min and it's actually 25 min, two bad
  things happen:
    - You dispatch a rider at minute 10 (to arrive at minute 15),
      rider waits 10 min at the seller. Wasted capacity.
    - You quoted the client 30 min and deliver at 40. Broken promise.

  If you estimate 25 min and it's actually 25 min:
    - You dispatch at minute 20. Rider waits 1 min. Efficient.
    - You quoted 40 min and deliver at 38. Client is pleasantly
      surprised.

  Prediction model (no ML needed):
    predicted_prep(seller, items, now) =
        seller.base_prep_time                    (from seller profile)
      + seller.per_item_increment × item_count   (bigger orders take longer)
      + seller.current_queue_depth × 2.5 min     (backlog penalty)
      + time_of_day_adjustment(seller, now)       (empirical curve)

  Calibration: after every delivery, compare food_ready_est vs
  actual_pickup_time. Compute error. Update seller.base_prep_time
  as an exponential moving average with α=0.05. The system LEARNS
  each seller's rhythm without any ML infrastructure. Just
  arithmetic and a feedback loop.

  This alone cuts T_pickup_wait by 40–60% in simulations because
  riders stop arriving 10 minutes before the food is ready.

  Time budget consumed at end of step ②: ~0 min (cooking hasn't
  finished yet, just started).

--------------------------------------------------------------------------------
5. STEP ③ — SPECULATIVE PRE-DISPATCH (new in v4)
--------------------------------------------------------------------------------
  This is the key innovation for elapsed-time optimization.

  In v2/v3 dispatch was DEFERRED: wait until close to food_ready,
  then find a rider. This minimizes rider idle time at the seller
  but ADDS time if no rider is nearby when food becomes ready.

  v4 introduces SPECULATIVE dispatch: tentatively reserve a rider
  EARLY, but don't commit them until a confirmation gate.

  Trigger: T_remaining_prep drops below speculative_horizon.
    speculative_horizon = max(rider_travel_to_seller) in quadrant
                        + buffer (2 min)
    Typically: ~10–12 min before food_ready_est.

  Actions:
    (a) Run the full Phase A + Phase B pipeline (v3 §3–5) for all
        orders crossing the horizon THIS tick.
    (b) For orders where a good match exists (cost < threshold):
          rider.state → SPECULATIVELY_RESERVED
          order.state → PRE_DISPATCHED
        Rider is NOT told to move yet. They continue current task
        or stay idle. They are simply excluded from OTHER orders'
        candidate pools.
    (c) For orders where no good match exists: do nothing. They'll
        enter the regular dispatch window at step ④.

  Why speculative:
    - Guarantees a rider EXISTS for this order before food is done.
    - Doesn't waste rider time: they don't move until step ④.
    - If conditions change (rider goes offline, better match
      appears), the reservation is REVOCABLE until step ④. This
      is the critical difference from v2/v3's commit-on-assignment.

  State after: PRE_DISPATCHED (revocable) or still PREPARING.

  Time budget consumed: still just T_prep (cooking in progress).

--------------------------------------------------------------------------------
6. STEP ④ — RIDER ASSIGNMENT (firm commit)
--------------------------------------------------------------------------------
  Trigger: T_remaining_prep ≤ rider_travel_to_seller + 90s (safety).
    This is the same deferred-dispatch window from v2/v3, but now
    most orders already have a speculative reservation.

  Actions:
    Two paths:

    PATH A — Order has a speculative reservation (typical, ~80%):
      (a) Re-validate: is the reserved rider still available and
          still the best choice? Quick check: compare reserved
          rider's current cost vs best alternative. If reserved
          rider is within 15% of best → CONFIRM reservation.
          If dramatically better option appeared → swap.
      (b) Confirm:
            rider.state → EN_ROUTE_TO_SELLER
            order.state → RIDER_ASSIGNED
          NOW the rider is told to move. Push notification with
          seller address.
      (c) Emit: RiderAssigned.

    PATH B — No reservation (shortage or cancellation):
      Run the standard pipeline from v3 §3–5. Assign and commit.

  Batching decision is finalized here:
    If rider is EN_ROUTE_TO_SELLER for order X, and order Y just
    hit the dispatch window, and Y's seller is within 400m of X's
    seller, and adding Y doesn't bust X's SLA → AMEND rider's
    trip: add Y to the batch. This is OPPORTUNISTIC BATCHING, only
    during the commit step, only when it doesn't delay existing
    assignments.

  State after: RIDER_ASSIGNED.
  Time budget consumed: T_prep (mostly done) + rider travel starting.

  ──────────────────────────────────────────────────────────────────
  WHAT WE OPTIMIZED HERE
  ──────────────────────────────────────────────────────────────────
  Speculative pre-dispatch (③) + firm commit (④) creates a
  TWO-STAGE COMMIT that decouples "ensure a rider exists" from
  "send the rider." This eliminates the tradeoff from v2/v3:

    v2/v3: dispatch early → rider waits at seller (bad)
           dispatch late  → no rider available (worse)
    v4:    reserve early, dispatch late → rider doesn't wait AND
           rider is guaranteed.

  Expected pickup-wait reduction: 30–50% vs v3's deferred-only.

--------------------------------------------------------------------------------
7. STEP ⑤ — RIDER TRAVELING TO SELLER
--------------------------------------------------------------------------------
  Trigger: rider receives push notification with seller address.

  Actions:
    (a) Rider app shows navigation to seller.
    (b) Rider GPS pings every 5 seconds → Tracking module.
    (c) System continuously computes rider ETA to seller:
          eta_to_seller = manhattan(rider.pos, seller.pos) / 233 m/min
    (d) Client app shows "rider on the way to restaurant" with
        live position.
    (e) If rider deviates or stalls (eta_to_seller increases for
        3 consecutive pings), flag as potential issue. After 90s
        of stall → reassignment cascade (v3 §7).

  Live ETA recalculation:
    The client's quoted delivery ETA updates in real time:
      updated_eta = max(food_ready_est, rider_eta_to_seller)
                  + pickup_wait_est
                  + manhattan(seller, client) / 233

    If updated_eta > original quoted ETA + 5 min → proactive
    notification: "Your order is running a few minutes late."
    Expectation management BEFORE the client gets angry.

  State after: still RIDER_ASSIGNED, rider physically moving.
  Time budget consumed: T_prep (finishing) + T_rider_to_seller
  (in progress, overlapping with prep).

  ──────────────────────────────────────────────────────────────────
  THE OVERLAP IS THE WHOLE POINT
  ──────────────────────────────────────────────────────────────────
  When dispatch timing is perfect:
    T_prep and T_rider_to_seller run IN PARALLEL.
    Rider arrives at seller the moment food is ready.
    Effective contribution to T_total: max(T_prep, T_rider_to_seller)
    instead of T_prep + T_rider_to_seller.

  This is why prep-time prediction (§4) is the highest-impact
  optimization. Better prediction → better synchronization →
  the two longest phases overlap more → T_total drops.

--------------------------------------------------------------------------------
8. STEP ⑥ — FOOD READY + RIDER ARRIVES (SYNCHRONIZATION POINT)
--------------------------------------------------------------------------------
  This is a RENDEZVOUS. Two independent processes — cooking and
  rider travel — must converge. Three possible orderings:

  CASE A: Rider arrives first (early by Δ minutes).
    Rider waits at seller. This is T_pickup_wait.
    Cost: rider time wasted. Δ goes into pickup_wait_actual.
    If Δ > 5 min → prediction was bad. Log it, update seller's
    prep model (§4 calibration).

  CASE B: Food ready first (early by Δ minutes).
    Food sits on counter. This is also T_pickup_wait from the
    food's perspective.
    Cost: food quality degrades. Δ > 3 min → seller complaint.
    If Δ > 5 min → dispatch was late. Log it, consider lowering
    the safety margin for this seller.

  CASE C: Simultaneous (|Δ| < 1 min). IDEAL.
    T_pickup_wait ≈ 1 min (just the handoff).

  Actions at rendezvous:
    (a) Seller marks "food ready" in seller app OR system infers
        from prep-time model.
    (b) Rider arrives (GPS within 50m of seller for >30s).
    (c) Whichever is second triggers: order.state → READY_FOR_PICKUP.
    (d) Start the pickup clock.

  State after: READY_FOR_PICKUP.
  Time budget consumed: max(T_prep, T_rider_to_seller) +
  T_pickup_wait (goal: <2 min).

  Metric to obsess over: SYNCHRONIZATION ERROR = |Δ|.
  This is THE proxy metric for pipeline health. Plot it per seller,
  per hour. Improve prediction → Δ shrinks → everything else
  follows.

--------------------------------------------------------------------------------
9. STEP ⑦ — PICKUP
--------------------------------------------------------------------------------
  Trigger: order state = READY_FOR_PICKUP.

  Actions:
    (a) Rider picks up bag(s). For batches, may collect from 1–2
        sellers (if batch has 2 pickup stops, second stop comes now).
    (b) Rider taps "picked up" in rider app. If not confirmed
        within 4 min of READY_FOR_PICKUP → prompt rider, alert ops.
    (c) order.state → PICKED_UP. picked_up_at = now.
    (d) For batched orders: optimize drop sequence.
        Brute-force permutation of ≤4 drop points, pick the route
        that minimizes WORST individual SLA slack (not total
        distance — protect the tightest order first).
    (e) Rider app shows navigation to first client.
    (f) Client app switches to "rider has your food" with live map.
    (g) Emit: OrderPickedUp.

  State after: PICKED_UP.
  Time budget consumed: max(T_prep, T_rider_to_seller) +
  T_pickup_wait + handoff (~1 min).

--------------------------------------------------------------------------------
10. STEP ⑧ — RIDER TRAVELING TO CLIENT
--------------------------------------------------------------------------------
  Trigger: rider taps "picked up" / starts moving away from seller.

  Actions:
    (a) Rider app shows navigation to client (or first drop in
        batch).
    (b) GPS tracking continues. Client sees live rider position.
    (c) Continuous ETA update:
          eta_to_client = route_remaining / 233 m/min
    (d) When eta_to_client < 3 min → push to client: "Rider is
        almost there!" Gives client time to come downstairs / open
        door.
    (e) For batched deliveries:
        After each drop, recalculate route for remaining drops.
        Update remaining clients' ETAs.
        If a subsequent drop's SLA is in danger due to delays at
        an earlier drop → proactive notification.

  Anomaly detection:
    - Rider stationary for >3 min mid-route → prompt: "Are you
      okay? Need help finding the address?"
    - Rider moving away from client → rerouting prompt.
    - Rider speed < 5 km/h for sustained period → traffic jam,
      adjust ETA.

  State after: still PICKED_UP, rider physically moving.
  Time budget consumed: everything so far + T_travel (in progress).

--------------------------------------------------------------------------------
11. STEP ⑨ — DELIVERY
--------------------------------------------------------------------------------
  Trigger: rider GPS within 80m of client for >20s, OR rider taps
  "arrived."

  Actions:
    (a) Client notified: "Your rider is here."
    (b) Rider delivers bag. Client confirms in app (or rider takes
        photo if contactless).
    (c) If client unreachable: 5-min timer. Call client. If still
        no response: rider marks "client unavailable," moves to
        next drop (if batched) or goes idle. Ops handles refund.
    (d) order.state → DELIVERED. delivered_at = now.
    (e) Capture payment (was authorized in step ①).
    (f) Compute actual T_total = delivered_at − placed_at.
    (g) Emit: OrderDelivered.
    (h) FEEDBACK LOOP — update models:
        - Seller prep model: calibrate with actual prep time.
        - Rider speed model: calibrate with actual travel time.
        - Pickup wait model: calibrate with actual rendezvous Δ.
        All via exponential moving average — no ML, just fast,
        stable feedback.
    (i) If batched: rider proceeds to next client. Repeat ⑧–⑨.
    (j) After final drop:
        rider.state → IDLE.
        Suggest repositioning point (v3 §6.4).
        Rider enters candidate pool for next dispatch tick.

  State after: DELIVERED.

  Final T_total breakdown (optimized target):
    ┌──────────────────────────┬───────────┬─────────────────────┐
    │ Segment                  │ v3 (was)  │ v4 (target)         │
    ├──────────────────────────┼───────────┼─────────────────────┤
    │ T_queue (①→②)           │  0–1 min  │  0–1 min            │
    │ T_prep (②→⑥)            │  15 min   │  15 min (given)     │
    │ T_rider_to_seller (⑤→⑥) │  7 min    │  7 min (parallel w/ │
    │                          │  (serial) │  prep, so +0 net)   │
    │ T_pickup_wait (⑥→⑦)     │  4 min    │  1.5 min (synced)   │
    │ T_travel_to_client (⑧→⑨)│  5.5 min  │  5.5 min (given)    │
    ├──────────────────────────┼───────────┼─────────────────────┤
    │ T_total                  │ ~31 min   │ ~23 min             │
    └──────────────────────────┴───────────┴─────────────────────┘

  The 8-minute win comes almost entirely from:
    - Parallelizing prep and rider travel (spec pre-dispatch)
    - Shrinking pickup wait via better synchronization
  NOT from a fancier algorithm.

12. THE OPTIMIZATION FEEDBACK SYSTEM
================================================================================
  The pipeline isn't optimized by a smart algorithm at one point.
  It's optimized by a FEEDBACK LOOP that tightens every segment
  continuously.

    ┌──────────────┐      ┌──────────────┐      ┌──────────────┐
    │  PREDICT     │─────►│  EXECUTE     │─────►│  MEASURE     │
    │  prep time   │      │  the pipeline│      │  actual time │
    │  travel time │      │  steps ①–⑨   │      │  per segment │
    │  demand      │      │              │      │              │
    └──────┬───────┘      └──────────────┘      └──────┬───────┘
           │                                           │
           │         ┌──────────────┐                  │
           └─────────│  CALIBRATE   │◄─────────────────┘
                     │  update per- │
                     │  seller, per-│
                     │  rider, per- │
                     │  hour models │
                     └──────────────┘

  What gets calibrated:
    - seller.base_prep_time (per seller, α=0.05 EMA)
    - seller.time_of_day_curve (per seller per hour)
    - rider.avg_speed (per rider, weather-adjusted)
    - pickup_wait.by_seller (per seller)
    - demand_forecast.by_hour (global + per quadrant)

  Every order that completes step ⑨ feeds back into the prediction
  models used in step ① and ③. The system gets more accurate every
  day WITHOUT any ML deployment, WITHOUT a training pipeline, WITHOUT
  a data team.

  THIS is what moves the needle, not a heavier algorithm.
