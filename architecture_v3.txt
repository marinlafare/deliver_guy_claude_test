================================================================================
FOODY — ASSIGNMENT ALGORITHM ARCHITECTURE v3
(revised for corrected operating parameters)
================================================================================

--------------------------------------------------------------------------------
0. A RESPECTFUL CORRECTION BEFORE WE START
--------------------------------------------------------------------------------
  "We can't use a greedy algorithm" — v2 was NOT greedy. v2 used the
  Hungarian algorithm: the provably OPTIMAL solution to bipartite
  matching. There is no better single-order assigner.

  But your instinct is right that v2 breaks under these numbers. It
  just breaks for a different reason. The problem isn't greedy vs
  optimal. The problem is that NO single-order assignment — greedy,
  Hungarian, or handed down by God — can make 95 riders deliver 5,000
  orders a day one bag at a time. The math doesn't close.

  v2's algorithm was correct. v2's PROBLEM FORMULATION is now wrong.

--------------------------------------------------------------------------------
1. THE MATH — WHERE IT ACTUALLY BREAKS
--------------------------------------------------------------------------------
  Area is 6 km² → model as a 2449 × 2449 m square (integer coords).
  Speed dropped 18 → 14 km/h (= 233 m/min, was 300). A 22% haircut.

  1.1 Single-order cycle time
      rider → seller   avg ~1.6 km Manhattan ÷ 233 m/min  ≈  6.9 min
      pickup wait                                          =  4.0 min
      seller → client  avg ~1.3 km (radius-capped)        ≈  5.6 min
      handoff + reposition slack                           ≈  1.5 min
      ──────────────────────────────────────────────────────────────
      CYCLE                                                ≈ 18.0 min

  1.2 Fleet throughput ceiling (single-carry)
      95 riders × (60 / 18) deliveries/hr  =  ~317 orders/hour MAX
      (and that assumes zero idle time — never true)

  1.3 Demand
      5,000 / day over ~12 active hours    =  ~417 orders/hour AVG
      Peak (lunch/dinner, ~3× avg)         = ~1,250 orders/hour

  1.4 The gap
      ┌──────────────┬───────────┬──────────┬────────────┐
      │              │  Demand   │ Capacity │  Deficit   │
      ├──────────────┼───────────┼──────────┼────────────┤
      │ Average hour │    417    │   ~317   │   −24%     │
      │ Peak hour    │  1,250    │   ~317   │   −75%     │
      └──────────────┴───────────┴──────────┴────────────┘

      You are underwater even on an AVERAGE hour. At peak you can
      serve one in four orders. No assignment algorithm fixes this —
      it's a physics problem, not a scheduling problem.

  1.5 Geometry is now a real constraint too
      Worst-case rider→seller leg: opposite corners of a 2.45 km
      square = ~4.9 km Manhattan = 21 min at 14 km/h.
      Travel budget (45 − 15 − 4) = 26 min. A badly-placed rider
      eats 80% of the budget just GETTING to the seller. In v1/v2
      geometry was slack. Now it bites.

--------------------------------------------------------------------------------
2. THE ARCHITECTURAL SHIFT: SINGLE-CARRY → BATCHED DELIVERY
--------------------------------------------------------------------------------
  The only lever that closes the gap without hiring 200 more riders
  is putting MULTIPLE ORDERS IN ONE BAG.

  Batching factor vs throughput:
      1 order/trip   →  ~317/hr   (baseline, underwater)
      2 orders/trip  →  ~520/hr   (covers average, not peak)
      3 orders/trip  →  ~680/hr   (covers mild peak)
      dynamic 1–4    →  ~750/hr   (target; covers realistic peak
                                   after demand shaping, see §8)

  Why batching doesn't scale linearly: each extra stop adds routing
  overhead, and later stops in the batch eat more of their SLA. A
  batch of 4 is only ~2.4× the throughput of a single, not 4×.

  This changes the PROBLEM CLASS:

    v2:  order ↔ rider            bipartite matching   →  Hungarian
    v3:  {orders} ↔ rider         set partition +       →  two-phase
                                  routing + matching       heuristic

  v3 is a restricted Vehicle Routing Problem with Time Windows
  (VRPTW). VRPTW is NP-hard. We will not solve it optimally. We will
  solve it WELL and FAST with a structured heuristic, because the
  tick runs every 2 seconds and a 1% better solution that takes 30
  seconds is worthless.

--------------------------------------------------------------------------------
3. THE NEW PIPELINE — TWO-PHASE ASSIGNMENT
--------------------------------------------------------------------------------
  v2's pipeline survives but gains a phase. Still pure-functional
  core, still transactional shell.

    ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
    │ 1. COLLECT   │──►│ 2. FILTER    │──►│ 3a. BATCH    │──┐
    │  candidates  │   │  hard rules  │   │   FORMATION  │  │
    └──────────────┘   └──────────────┘   └──────────────┘  │
                                                            ▼
    ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
    │ 6. OBSERVE   │◄──│ 5. COMMIT    │◄──│ 4. SOLVE     │◄─┐
    │ decision log │   │  atomic txn  │   │  hungarian   │  │
    └──────────────┘   └──────────────┘   │  on BATCHES  │  │
                                          └──────────────┘  │
                                                            │
                                          ┌──────────────┐  │
                                          │ 3b. BATCH    │──┘
                                          │   COSTING    │
                                          └──────────────┘

  Phase A (3a): group orders into candidate BATCHES.
  Phase B (3b + 4): assign BATCHES to riders. This is bipartite
  again — Hungarian still applies, just with batches on the left
  side instead of orders. v2's solver is reused intact.

  The hard part moved from "matching" to "grouping."

--------------------------------------------------------------------------------
4. PHASE A — BATCH FORMATION
--------------------------------------------------------------------------------
  A batch is 1–4 orders that one rider will pick up and deliver in a
  single trip. The pickup route and drop route are fixed at batch
  creation.

  4.1 COMPATIBILITY RULES (hard — an order pair is batchable IFF)

      BC1  PICKUP PROXIMITY
           Sellers are the same, OR within 400 m Manhattan of each
           other. Rider makes at most 2 pickup stops; we don't want
           them zig-zagging across the zone collecting bags.

      BC2  DROP PROXIMITY
           Clients are within 700 m Manhattan of each other. Keeps
           the drop loop short enough that the last client's food
           isn't cold.

      BC3  TIME-WINDOW OVERLAP
           food_ready times within 5 min of each other. Rider can't
           wait 12 min at the seller for the second bag while the
           first one congeals.

      BC4  PER-ORDER SLA FEASIBILITY
           For EACH order in the proposed batch, simulate the full
           route (pickup sequence + drop sequence) and verify that
           THAT order's delivery time ≤ placed_at + 45 min. One
           order in a batch of 3 can be infeasible while the other
           two are fine — the whole batch is then invalid.

      BC5  THERMAL COMPATIBILITY
           Don't batch ice cream with hot soup. A simple tag on menu
           items (hot / cold / ambient) and a rule: a batch may
           contain at most two thermal classes, and "frozen" never
           rides with "hot" for more than 6 min.

  4.2 FORMATION ALGORITHM — GREEDY CLUSTERING
      (Yes, THIS part is greedy. The combinatorics force it.)

      Input: orders in the dispatch window, sorted by SLA-slack
             ascending (most urgent first).

      1. Seed: pop the most urgent order → new batch B = {o₁}.
      2. Grow: scan remaining orders for the one that
           (a) passes all compatibility rules BC1–BC5 with every
               order already in B, AND
           (b) minimizes the MARGINAL ROUTE COST:
                 Δ = route_time(B ∪ {o}) − route_time(B)
         Add it if Δ < marginal_threshold (≈ 4 min). Repeat until
         |B| = 4 or no compatible order exists.
      3. Emit B. Goto 1 until order pool empty.

      Route time for a small batch (≤4 stops) is computed by brute-
      force permutation: 4 pickups → at most 4! = 24 orderings, same
      for drops. 24 × 24 = 576 route candidates, each is a handful
      of Manhattan-distance sums. Microseconds. No need for real TSP.

      Why seed with most-urgent: urgent orders get first pick of
      batch-mates and can grab a batch slot even as a singleton.
      Relaxed orders fill in around them.

  4.3 DUAL CANDIDACY
      An order may appear in multiple candidate batches (as a
      singleton AND as part of a pair with its neighbor). The
      Hungarian solver in phase B will pick the configuration with
      lowest total cost. We generate up to 3 candidate batches per
      order, tagged with a mutual-exclusion group so the solver
      can't assign two batches that share an order. (Implemented as
      a post-solve conflict check + one repair pass — cheaper than
      encoding exclusion into the LP.)

--------------------------------------------------------------------------------
5. PHASE B — BATCH-TO-RIDER ASSIGNMENT
--------------------------------------------------------------------------------
  This IS v2 with "order" replaced by "batch." The cost function
  gains batch-aware terms; the Hungarian solver is unchanged.

  5.1 COST TERMS (extends v2 §4)

      ────────────────────────────────────────────────────────────────
      TERM: BATCH ROUTE TIME                          weight: w_route
      ────────────────────────────────────────────────────────────────
        cost_route = rider_to_first_pickup
                   + pickup_loop
                   + last_pickup_to_first_drop
                   + drop_loop
        Replaces v2's simple two-leg distance. Computed per rider
        because rider_to_first_pickup depends on rider position.

      ────────────────────────────────────────────────────────────────
      TERM: WORST-ORDER SLA                           weight: w_sla
      ────────────────────────────────────────────────────────────────
        cost_sla = exp(−min_slack_in_batch / 5 min)
        The batch is only as good as its worst-served order. Use the
        MINIMUM slack across all orders in the batch, not the average.
        A batch where 3 orders have 15 min slack and one has 90
        seconds is a bad batch — the exponential makes sure the
        solver knows it.

      ────────────────────────────────────────────────────────────────
      TERM: BATCH EFFICIENCY BONUS                    weight: w_eff
      ────────────────────────────────────────────────────────────────
        cost_eff = −(|batch| − 1) × efficiency_bonus
        Negative cost for multi-order batches. Without this the
        solver would happily assign everything as singletons (lowest
        per-order cost, catastrophic for throughput). This term is
        the system TELLING ITSELF that throughput matters.
        Tune high during peak, near-zero off-peak.

      ────────────────────────────────────────────────────────────────
      TERMS CARRIED FROM v2 (adapted)
      ────────────────────────────────────────────────────────────────
        w_sync   — rider arrival vs EARLIEST food_ready in batch
        w_fair   — unchanged (per rider, not per order)
        w_chain  — unchanged (rider's current drop near batch's
                   first pickup)
        w_acc    — unchanged

  5.2 SOLVER
      Hungarian on cost matrix C[batches][riders]. Instance size:
      even at peak, ~60 candidate batches × ~35 available riders.
      Still O(n³) on tiny n. Still microseconds.

      Post-solve conflict resolution: if two assigned batches share
      an order (dual candidacy, §4.3), keep the lower-cost
      assignment, dissolve the other batch back into singletons,
      re-run Hungarian on the residual. Converges in ≤2 passes in
      practice because conflicts are rare.

--------------------------------------------------------------------------------
6. SPATIAL REVISIONS — GEOMETRY NOW HAS TEETH
--------------------------------------------------------------------------------
  6.1 Coordinate system
      Integer meters, 0..2448 on each axis. Same as before, bigger
      range.

  6.2 Bucket grid
      250 m buckets → 10 × 10 = 100 buckets. 200 sellers ≈ 2 per
      bucket. Still O(1) neighbor queries.

  6.3 SOFT ZONING (new)
      Divide the 2449 m square into 4 quadrants (NW, NE, SW, SE),
      ~1.2 km each side. Each rider has a HOME quadrant.

        cost_zone = w_zone × (rider.home_quadrant ≠ batch.centroid_quadrant
                              ? quadrant_distance : 0)

      This is a SOFT preference, not a hard rule. A rider CAN cross
      quadrants when SLA pressure demands it, but the cost function
      nudges them to stay local. Effect: riders naturally drift back
      toward their home zone between deliveries, which keeps average
      rider→seller distance down. In v1/v2 this didn't matter (zone
      was small, riders were fast). Now a rider stranded in the
      wrong corner is 21 min from being useful.

  6.4 IDLE REPOSITIONING (new)
      When a rider goes IDLE, the system suggests a "park here"
      point: the demand-weighted centroid of their home quadrant's
      sellers, adjusted by time-of-day order heatmap. Rider app
      shows it as a soft suggestion. Compliance is optional but
      incentivized (idle riders at suggested spots get a small
      cost_position bonus on their next assignment).

  6.5 The 2 km delivery radius — an ally
      Clients can only order from sellers within 2 km. This CAPS the
      seller→client leg at ~8.6 min. The uncontrolled leg is
      rider→seller. Zoning (§6.3) and repositioning (§6.4) exist to
      tame that leg.

7. HARD CONSTRAINTS — REVISED
--------------------------------------------------------------------------------
  v2's HC1–HC5 survive. Add:

    HC6  PER-ORDER SLA IN BATCH CONTEXT
         For each order o in batch B assigned to rider r:
           simulated_delivery(o | B, r) ≤ o.placed_at + 45 min
         This is checked per (batch, rider) edge, not just per
         batch — the same batch may be feasible for a nearby rider
         and infeasible for a distant one.

    HC7  BATCH THERMAL WINDOW
         The longest any single bag rides in the batch ≤ 18 min.
         (From pickup of that bag to drop of that bag.) Food quality
         floor independent of SLA.

    HC8  RIDER CAPACITY
         |batch| ≤ rider.bag_capacity. Most riders: 3. Riders with
         the big insulated box: 4. Stored per rider.

8. DEMAND SHAPING — WHEN EVEN BATCHING ISN'T ENOUGH
--------------------------------------------------------------------------------
  Batching gets you to ~750/hr. True peak is ~1,250/hr. There will
  be 60–90 min windows where demand exceeds batched capacity. The
  algorithm cannot conjure riders. The BUSINESS must respond. The
  architecture must give the business the levers.

  8.1 PREDICTIVE THROTTLE
      A rolling 10-min forecast (EWMA of order rate + day-of-week +
      hour seasonality — no ML, just arithmetic) predicts demand for
      the next 30 min. When forecast > capacity × 0.9:

  8.2 LEVER 1 — HONEST ETAs
      Quoted delivery time shown to client at checkout inflates to
      the TRUE expected time (could be 55, 65 min). Some clients
      self-select out. This is the gentlest lever; pull it first.

  8.3 LEVER 2 — RADIUS CONTRACTION
      Temporarily shrink the effective delivery radius from 2 km to
      1.5 km. Shorter seller→client legs → faster cycles → higher
      throughput. Clients at the fringe see fewer seller options.
      Auto-reverts when forecast drops.

  8.4 LEVER 3 — SELLER-SIDE THROTTLE
      Sellers with current prep-time > 20 min get temporarily hidden
      from search. They're already a bottleneck; don't pile on.

  8.5 LEVER 4 — HARD CAP
      If the AWAITING_RIDER queue depth exceeds N, new checkouts are
      blocked with "we're slammed, try again in 15 min." Brutal but
      honest. A rejected order is better than a 90-min delivery and
      a 1-star review.

  These are BUSINESS decisions surfaced as CONFIG. The architecture
  provides the signals (forecast, queue depth) and the knobs (radius,
  ETA inflation, cap). Ops decides which to pull.

  latency. Over capacity, batching is existential.

--------------------------------------------------------------------------------
12. WHAT TO TELL YOUR BOSS
--------------------------------------------------------------------------------
  The parameter correction didn't reveal a flaw in the algorithm.
  It revealed a flaw in the FLEET SIZE. 95 riders at 14 km/h serving
  5,000 orders/day is physically under-provisioned by about 30% on
  average and 75% at peak.

  Software can claw back most of that gap via batching (this doc).
  It cannot claw back all of it. The honest conversation is:

    (a) Accept demand shaping at peak (some orders turned away), OR
    (b) Hire ~40 more riders, OR
    (c) Get faster vehicles (e-bikes: 14 → 22 km/h closes most of
        the gap on its own).

  The architecture supports all three. It cannot choose for you.
