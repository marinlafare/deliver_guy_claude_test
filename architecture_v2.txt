================================================================================
FOODY — ASSIGNMENT ALGORITHM ARCHITECTURE v2
================================================================================

--------------------------------------------------------------------------------
0. FRAMING THE PROBLEM CORRECTLY
--------------------------------------------------------------------------------
  This is NOT "find the nearest rider."
  This is MINIMUM-COST BIPARTITE MATCHING under constraints.

    Left set  : orders that need a rider in the next ~2 min
    Right set : riders who are (or are about to be) free
    Edge cost : how bad is THIS pairing?
    Goal      : pick a set of pairings that minimizes TOTAL cost

  Why this framing matters:
    Greedy nearest-rider is locally optimal but globally wrong.
    Classic failure:
        Rider A is 200m from order 1 AND 200m from order 2.
        Rider B is 250m from order 1 but 900m from order 2.
        Greedy: A→1 (A is closest), then B→2. Total = 200+900 = 1100m.
        Optimal: B→1, A→2.                     Total = 250+200 =  450m.
    At 300 orders/hour this mistake compounds into SLA breaches.

  Instance size at peak:
    ~15 orders in dispatch window × ~35 available riders.
    Hungarian algorithm: O(n³) ≈ 40k ops. Runs in microseconds.
    We can afford the OPTIMAL solution. So we compute it.

--------------------------------------------------------------------------------
1. ARCHITECTURE: A 5-STAGE PIPELINE
--------------------------------------------------------------------------------
  The assignment engine is a pure function wrapped in a transactional
  shell. Runs every dispatcher tick (2s).

    ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
    │ 1. COLLECT   │──►│ 2. FILTER    │──►│ 3. COST      │──┐
    │  candidates  │   │  hard rules  │   │  matrix      │  │
    └──────────────┘   └──────────────┘   └──────────────┘  │
                                                            ▼
    ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
    │ 6. OBSERVE   │◄──│ 5. COMMIT    │◄──│ 4. SOLVE     │
    │ decision log │   │  atomic txn  │   │  hungarian   │
    └──────────────┘   └──────────────┘   └──────────────┘

  Design rule: stages 1–4 are PURE (no side effects, no I/O).
  They take a snapshot of world state and return a proposed plan.
  Only stage 5 touches the database. This makes the core algorithm
  trivially unit-testable: feed it a synthetic world, assert on the
  plan it produces.

--------------------------------------------------------------------------------
2. STAGE 1 — CANDIDATE COLLECTION
--------------------------------------------------------------------------------
  2.1 Which ORDERS enter the pool this tick?

      An order becomes dispatchable when a rider leaving NOW would
      arrive roughly when the food is ready.

        dispatch_window_open  = food_ready_est − max_rider_travel
                                               − safety_margin (90s)
        dispatch_window_close = food_ready_est + max_late_tolerance (3 min)

      Pool = orders WHERE state = AWAITING_RIDER
               AND now ∈ [dispatch_window_open, dispatch_window_close]
             ORDER BY priority DESC, placed_at ASC

      Orders past dispatch_window_close are STARVING → forced into
      pool with priority=CRITICAL regardless of rider fit.

  2.2 Which RIDERS enter the pool this tick?

      Three classes, all eligible:

      ┌─────────────┬──────────────────┬─────────────────────────────┐
      │ Class       │ Available at     │ Available from              │
      ├─────────────┼──────────────────┼─────────────────────────────┤
      │ IDLE        │ now              │ current GPS position        │
      │ DROPPING    │ now + eta_drop   │ client's position           │
      │ RETURNING   │ now              │ current GPS position        │
      └─────────────┴──────────────────┴─────────────────────────────┘

      Including DROPPING riders is the CHAINING optimization.
      A rider 90 seconds from finishing a delivery, whose drop-off
      point happens to be near the next seller, is often the best
      pick — better than an idle rider across the zone. This is
      where peak-hour throughput comes from.

      We model each rider as a (free_at_time, free_at_position) tuple.
      For IDLE that's (now, gps). For DROPPING it's a prediction.

--------------------------------------------------------------------------------
3. STAGE 2 — HARD CONSTRAINTS (FILTER, NOT WEIGH)
--------------------------------------------------------------------------------
  Hard constraints are BOOLEAN. They remove edges from the graph
  entirely. They are never "outweighed" by a good score elsewhere.

    HC1  SLA FEASIBILITY
         rider_free_at + travel_to_seller + pickup_wait
                       + travel_to_client  ≤  order.placed_at + 45 min
         If this fails, the pairing is physically impossible. Cut it.

    HC2  REJECTION HISTORY
         Rider already rejected this specific order → cut.
         (Prevents offer/reject ping-pong.)

    HC3  SHIFT BOUNDARY
         Estimated delivery completion > rider.shift_end → cut.
         Don't strand a rider past their clock-out.

    HC4  CAPACITY
         If the system ever supports order batching (one rider, two
         bags), a rider at max capacity is cut for new singles.

    HC5  SELLER BLACKLIST
         Seller has flagged this rider (or vice versa) → cut.
         Rare, but ops needs this lever.

  After filtering, some orders may have ZERO candidate riders.
  These are set aside (see §7 escalation) and excluded from the
  matching — they must not distort the optimization for the rest.

--------------------------------------------------------------------------------
4. STAGE 3 — THE COST FUNCTION (THE SOUL OF THE SYSTEM)
--------------------------------------------------------------------------------
  For every surviving (order, rider) edge, compute a scalar cost.
  Lower = better. The cost is a weighted sum of independent TERMS.

  ──────────────────────────────────────────────────────────────────
  4.1 TERM: ARRIVAL SYNC                               weight: w_sync
  ──────────────────────────────────────────────────────────────────
      Δ = rider_arrival_at_seller − food_ready_time

      cost_sync = │ Δ × 1.0    if Δ < 0   (rider early → rider waits)
                  │ Δ × 2.5    if Δ ≥ 0   (rider late  → food cools,
                  │                         client waits)

      ASYMMETRIC by design. A rider idling 3 min at the seller is
      annoying. Food sitting 3 min on the counter is a quality hit
      AND eats the SLA budget. Punish lateness harder.

  ──────────────────────────────────────────────────────────────────
  4.2 TERM: TOTAL LEG DISTANCE                         weight: w_dist
  ──────────────────────────────────────────────────────────────────
      cost_dist = manhattan(rider_free_pos, seller)
                + manhattan(seller,         client)

      The second leg is fixed per order (doesn't depend on rider
      choice) BUT including it lets the solver prefer giving the
      long-delivery order to the rider who's already close to the
      seller, and the short-delivery order to the rider further out.
      Total system travel drops.

  ──────────────────────────────────────────────────────────────────
  4.3 TERM: SLA PRESSURE                               weight: w_sla
  ──────────────────────────────────────────────────────────────────
      slack = (order.placed_at + 45 min) − predicted_delivery_time

      cost_sla = exp(−slack / 5 min)

      Exponential: near-zero when slack is comfy (>15 min), explodes
      as slack approaches zero. An order with 4 min of slack will
      outbid almost anything for the nearest rider. This is how
      starving orders get rescued without special-case code.

  ──────────────────────────────────────────────────────────────────
  4.4 TERM: FAIRNESS                                   weight: w_fair
  ──────────────────────────────────────────────────────────────────
      cost_fair = rider.deliveries_today − fleet_median_today

      Riders above median get a positive penalty, below get a bonus.
      Prevents the "golden corner" problem where one rider parked at
      a hot seller cluster scoops every order while others starve.
      Riders are paid per delivery; perceived unfairness causes churn.

      Weight this LOW during peak (throughput matters more), HIGHER
      during off-peak (plenty of slack, be equitable). See §8.

  ──────────────────────────────────────────────────────────────────
  4.5 TERM: CHAIN QUALITY (bonus, negative cost)       weight: w_chain
  ──────────────────────────────────────────────────────────────────
      Applies only to riders in DROPPING state.

      cost_chain = −max(0,  bonus_cap
                          − manhattan(rider_drop_pos, seller))

      Rewards assignments where the rider's current delivery ends
      near the next pickup. This is the mechanism that turns 95
      independent riders into something resembling a coordinated
      fleet. Cap the bonus so it can't override SLA pressure.

  ──────────────────────────────────────────────────────────────────
  4.6 TERM: ACCEPTANCE RISK                            weight: w_acc
  ──────────────────────────────────────────────────────────────────
      cost_acc = (1 − rider.acceptance_rate_7d) × reassignment_penalty

      A rider who rejects 30% of offers costs you a re-dispatch cycle
      30% of the time. Price that in. This naturally routes orders
      toward reliable riders during crunch without any explicit
      "reliability tier" system.

  ──────────────────────────────────────────────────────────────────
  FINAL COST
  ──────────────────────────────────────────────────────────────────
      C(order, rider) =   w_sync  × cost_sync
                        + w_dist  × cost_dist
                        + w_sla   × cost_sla
                        + w_fair  × cost_fair
                        + w_chain × cost_chain
                        + w_acc   × cost_acc
                        + ε × rider_id        ← determinism tiebreak

      All terms normalized to "equivalent minutes of delay" before
      weighting, so weights are dimensionless and intuitive to tune.

--------------------------------------------------------------------------------
5. STAGE 4 — SOLVER
--------------------------------------------------------------------------------
  Input : cost matrix  C[orders][riders], with +∞ for filtered edges.
  Output: a set of (order, rider) pairs, at most one per order, at
          most one per rider, minimizing Σ C.

  Algorithm: HUNGARIAN (Kuhn–Munkres).
    - Exact optimal solution for the assignment problem.
    - O(n³). n ≤ 40 here → finishes before you can blink.
    - Use scipy.optimize.linear_sum_assignment or equivalent; do not
      hand-roll this.

  Rectangular matrix handling:
    - More riders than orders (usual): unmatched riders stay idle.
      This is fine.
    - More orders than riders (peak crunch): pad with virtual riders
      at cost +∞. Unmatched orders roll to next tick. Priority/SLA
      pressure ensures the oldest ones win the next round.

  Why not greedy?  See §0. At this scale optimal is free.
  Why not ILP?    Overkill. No complex constraints that Hungarian
                  can't express. Would add a solver dependency and
                  100x latency for zero quality gain.
  Why not ML?     No training data on day one. See §10.

--------------------------------------------------------------------------------
6. STAGE 5 — COMMIT
--------------------------------------------------------------------------------
  The plan from stage 4 is a PROPOSAL. The world may have shifted in
  the milliseconds since the snapshot (rider went offline, order got
  cancelled). Commit defensively.

    BEGIN TRANSACTION
      FOR EACH (order, rider) IN plan:
        -- Optimistic concurrency check
        UPDATE riders  SET state='EN_ROUTE_TO_SELLER', assigned=order
          WHERE id=rider AND state IN ('IDLE','RETURNING')
        UPDATE orders  SET state='RIDER_ASSIGNED', rider=rider
          WHERE id=order AND state='AWAITING_RIDER'
        -- If either UPDATE hit 0 rows → skip this pair, don't abort
        -- the whole batch. Log it. It'll re-enter next tick.
        INSERT INTO order_events (order, 'RIDER_ASSIGNED', rider, ...)
        INSERT INTO outbox (rider_push_notification, ...)
    COMMIT

  Single transaction for the whole batch → atomic from the outside,
  but individual pair failures don't poison the batch. Best of both.

--------------------------------------------------------------------------------
7. REASSIGNMENT & ESCALATION LADDER
--------------------------------------------------------------------------------
  Assignment is an OFFER, not an order. Riders can reject (explicit)
  or go dark (implicit timeout: 30s).

    Attempt 1  → normal pool, normal cost.
    Attempt 2  → priority=HIGH. Adds a flat −5 min bonus to every
                 edge for this order, so it outbids fresh orders.
    Attempt 3  → priority=CRITICAL. Exempted from fairness + chain
                 terms. Only sync/dist/sla matter. Grab anyone.
    Attempt 4  → Human escalation. Page the ops dashboard. Possible
                 outcomes: manual assign, split order, proactive
                 client apology + voucher.

  A rider who goes dark is marked SUSPECT. Three dark events in an
  hour → auto-offline, needs manual re-activation. Protects the pool
  from a dead phone draining offers.

--------------------------------------------------------------------------------
8. POLICY vs MECHANISM — THE CRITICAL SEPARATION
--------------------------------------------------------------------------------
  MECHANISM (code, changes rarely):
    - The pipeline stages.
    - The Hungarian solver.
    - The shape of the cost function (which terms exist).

  POLICY (config, changes weekly):
    - The weights w_sync, w_dist, w_sla, w_fair, w_chain, w_acc.
    - Hard constraint thresholds (SLA minutes, shift buffer).
    - Dispatch window margins.

  Policy lives in a DB table, hot-reloaded every 60s. Ops can tune
  the fleet's behavior without a deploy. "Riders complaining about
  fairness" → bump w_fair from 0.3 to 0.5, watch the dashboard.

  POLICY PROFILES — weights are not global constants, they're
  functions of context:

    ┌────────────┬────────┬────────┬────────┬────────┬─────────┐
    │ Profile    │ w_sync │ w_dist │ w_sla  │ w_fair │ w_chain │
    ├────────────┼────────┼────────┼────────┼────────┼─────────┤
    │ off_peak   │  1.0   │  0.8   │  1.0   │  1.2   │   0.5   │
    │ peak       │  1.0   │  1.0   │  2.0   │  0.3   │   1.5   │
    │ rain       │  0.8   │  1.3   │  2.5   │  0.2   │   1.8   │
    │ understaff │  0.7   │  1.0   │  3.0   │  0.0   │   2.0   │
    └────────────┴────────┴────────┴────────┴────────┴─────────┘

  Profile selection is automatic (clock + active rider count + order
  rate) with manual override. During peak: fairness takes a back
  seat, SLA and chaining dominate. Off-peak: be nice to riders.

--------------------------------------------------------------------------------
9. OBSERVABILITY — THE DECISION LOG
--------------------------------------------------------------------------------
  Every assignment writes a DECISION RECORD. Not just "rider 42 got
  order 771" — the full reasoning.

    {
      tick_id, order_id, chosen_rider_id,
      policy_profile: "peak",
      chosen_cost: { total: 4.2, sync: 1.1, dist: 2.0, sla: 0.3, ... },
      runners_up: [
        { rider_id: 17, total: 5.8, sync: 0.2, dist: 4.9, ... },
        { rider_id: 03, total: 6.1, ... },
      ],
      rejected_by_hard_constraint: [
        { rider_id: 88, rule: "HC1_SLA", slack: -120s },
      ],
      outcome: null    -- filled in later: accepted? delivered_at? sla_met?
    }

  This is expensive to store (~2 KB/order × 2000/day = 4 MB/day,
  fine) and priceless to have:

    - Debugging: "Why did rider 42 get that order over rider 17?"
      → open the record, read the cost breakdown. 30-second answer.
    - Tuning: "Our w_fair is too high" → query records where the
      fairness term flipped the winner, check if SLA suffered.
    - ML training data: this IS the feature vector. When you're
      ready for §10, the dataset already exists.
    - Simulation: replay a day's snapshots through a new cost
      function, diff the plans, estimate impact before shipping.

--------------------------------------------------------------------------------
10. EVOLUTION PATH
--------------------------------------------------------------------------------
  Phase 1 (launch)    : Hand-tuned weights. One profile. Ship it.
  Phase 2 (month 2)   : Multiple profiles. Ops-tunable. A/B test
                        weights by splitting ticks 90/10.
  Phase 3 (month 6+)  : Replace the scalar cost function with a
                        learned model (gradient-boosted trees on the
                        decision-log features, predicting sla_met +
                        rider_accepted + client_rating). The PIPELINE
                        stays identical — only stage 3 swaps out. The
                        Hungarian solver still runs on top. This is
                        why the policy/mechanism split matters.
  Never               : Reinforcement learning. 2k orders/day is not
                        enough signal, and the failure modes during
                        exploration would torch your NPS.
